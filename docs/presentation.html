Model
02 / 10

Five visibility tiers

  • public - readable by anyone on the relay. No encryption.
  • group - NIP-29 relay-enforced groups. Scoped by h tag.
  • circle - ad-hoc participant sets. Deterministic hash of sorted pubkeys.
  • personal - user-auditable. NIP-44 self-encrypted. User + their agents can read.
  • internal - agent-only reasoning. NIP-44 self-encrypted. Invisible to users.
  • Topics - forward-slash namespaces: rust/error-handling, user/k0/preferences

Nostr relay is the canonical store. Local cache: SurrealDB (embedded, pure Rust) - HNSW vector search, BM25 full-text, and native graph edges in one engine. If local state is lost, nomen sync recovers everything from the relay.

Visibility is not binary. Scope is social and contextual.

Memory lives where its scope allows

personal / internal - NIP-44 self-encrypted, only author's agents
group - NIP-29 relay-enforced, scoped by h-tag
circle - ad-hoc participant set, hash of sorted pubkeys
public - plaintext, discoverable, trust-ranked

Groups form overlapping social boundaries. An npub in multiple groups accesses shared memory from each. Circles are ad-hoc - no relay group needed, participants listed in p-tags.

npubs are nodes. Groups are overlapping scopes. Public is the outer layer.

Intelligence
05 / 10

Memory intelligence layer

Three capabilities that turn flat memory into a structured, evolving knowledge system. All modular - each feature is independently toggle-able via config.

🕸️ Graph-aware retrieval

Search doesn't stop at direct hits. Traverses SurrealDB graph edges - mentions, references, contradicts - to surface related memories the query didn't explicitly match.

Enables: multi-hop reasoning, contradiction surfacing, entity-centric views

🧬 LLM entity extraction

During consolidation, LLM extracts entities (people, projects, concepts) and typed relationships between them: works_on, collaborates_with, decided, depends_on.

Enables: knowledge graph, relationship reasoning, richer graph retrieval

🧊 Cluster fusion

Groups related memories by topic namespace (user/k0/*, project/nomen/*) and synthesizes coherent cluster summaries via LLM.

Enables: coherent profiles, reduced noise, memory scalability

Design informed by "Memory in the Age of AI Agents" (Hu et al., 2025) - local consolidation with LLM merging, graph-structured memory (2D), and sleep-inspired dynamics. Each feature degrades gracefully: no API key → heuristic fallback or feature off.

Flat storage with graph metadata → structured, evolving knowledge.

Event model
06 / 10

Kind 31234 - addressable replaceable events

// Nomen memory event (kind 31234) { "kind": 31234, "pubkey": "<author-hex>", "created_at": 1739901000, "content": "{\"summary\":\"Use anyhow for app errors\", \"detail\":\"Prefer anyhow::Result…\", \"context\":null}", "tags": [ ["d", "public::rust-error-handling"], ["model", "anthropic/claude-opus-4-6"], ["confidence", "0.92"], ["version", "1"], ["t", "rust"], ["t", "error-handling"] ] }

D-tag format: {visibility}:{scope}:{topic}

  • public::rust-error-handling
  • group:techteam:deployment
  • circle:a3f8b2c1:shared-notes
  • personal:d29f…:ssh-config
  • internal:d29f…:agent-reasoning

Why visibility in d-tag?

  • Relay can filter by d-tag prefix: group:techteam:*
  • Same d-tag = replaceable (new version overwrites old)
  • No separate tier/scope tags needed
  • Content: JSON with summary, detail, context
Memory events are regular Nostr events with scope and topic semantics.

Integration
08 / 10

Three integration paths

MCP server

Protocol path. Expose search / add / publish / query as MCP tools. Works with any MCP-capable agent framework.

Best for: existing agent setups that already use MCP. Lowest integration friction.

Context-VM

Nostr-native path. Agent publishes kind 21900 requests, Nomen responds with kind 21901. All payloads NIP-44 encrypted. Also enables payments - memory access can be metered over Lightning.

Best for: Nostr-native agents. No transport dependency beyond relays. Supports paid memory services.

Library

Native embed path. Direct API for search, write, scope management, and trust-weighted retrieval. Deepest control.

Best for: custom agent implementations or apps needing tight memory integration.

Same memory core, same event model, same scoping rules. The integration surface changes, not the memory semantics.

Protocol, runtime, or native embed - choose what fits your stack.

Quickstart
10 / 10

Getting started

nomen init

Interactive setup: relay, keys, embedding provider, consolidation, server.

nomen store "rust/error-handling" \ --summary "Use anyhow for app errors" \ --detail "Prefer anyhow::Result…" \ --tier public --confidence 0.92

Store a memory. Published to relay as kind 31234.

nomen search "error handling" --graph nomen search "Alhovuori" --graph --hops 2

Hybrid search + graph traversal. --graph follows edges to surface related memories.

nomen consolidate nomen cluster --dry-run nomen entities --relations

Consolidate raw messages, preview cluster fusion, inspect entity relationships.

nomen serve --stdio nomen serve --http 127.0.0.1:3000

MCP server (stdio or HTTP). Tools: search, store, ingest, consolidate, entities, cluster, groups.

nomen serve --stdio --context-vm \ --allowed-npubs <hex-pubkey>

Context-VM: pure Nostr request/response (kind 21900→21901, NIP-44 encrypted).

CLI and server entry points for the workshop.
